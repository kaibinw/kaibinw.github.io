Household Dashboard

SC SUPPLYTK 03FPS 24TC 04:41:33:07
ROLE Product, design, engineering direction STACK Next.js · Notion · Home Assistant · Supabase STATUS In build, core views live

Four systems wired into one buying-decisions screen, with an agent doing the overnight legwork.

The studio problem

Ask a facility what it owns and where it's thin, and the answer lives in five places: floating license counts in a vendor portal (who's holding the Nuke seats right now?), render capacity in the farm manager, kit and drives in a spreadsheet someone updates quarterly, renewal dates in a producer's calendar, and pricing in whatever the reseller last quoted. Nobody sees "what's low, where, and what needs restocking" on one screen — so seats run out mid-crunch and renewals land as surprises.

A household has the identical problem with smaller numbers: consumables run out, wanted purchases wait for the right price, and the budget lives in a different system from the shopping list.

The same problem at household scale

Household procurement is scattered state: consumable stock lives in your head, the wants list lives in Notion, spend history lives in Ledger, and deals live on forums you'd have to check daily. Buying well means joining all four at the moment of decision, which nobody does at 9pm on a Tuesday.

The wiring

Stock stateHome Assistant tracks consumables; NFC tags on the shelf mark things low with one phone tap.
Wants listNotion database of things worth waiting for a deal on, with target prices.
Spend contextRead-only queries against Ledger's Supabase, so a purchase is seen against the month's budget.
Deal agentA scheduled Claude agent parses deal feeds and price-history data overnight, matching against the wants list and flagging genuine drops.

The NFC layer: physical world in, one tap

Stock state has to be effortless or it rots. So the shelf itself is the interface: an NFC tag on the pantry shelf, tapped with a phone, fires an Apple Shortcut into a Home Assistant webhook, which flips an input_boolean for that consumable. Automations hang off those booleans (low-stock notifications, the dashboard's buy list), and a dedicated Home Assistant dashboard is in build to show stock state natively alongside the house's energy and climate cards. The same pattern a studio would use for kit checkout: tag the shelf, not the person.

The data plumbing, proven elsewhere first

The wiring pattern isn't new to me: I run the same architecture in studio tooling that scans large delivery trees, tracks item state through a review-and-publish flow, and serves web-based UI panels off a small local HTTP server, with expensive sources loaded eagerly and everything else stubbed lazily so the first paint is instant. The dashboard reuses those exact moves: each source polled on its own cadence, one normalised feed, state transitions tracked rather than recomputed, and a kanban-style "needs a decision" queue as the primary surface. Different nouns, same plumbing.

What the morning view looks like

What it demonstrates

  • Agents as scheduled colleagues. The deal scan is a nightly agentic job with a narrow brief and a paper trail, the same pattern as an overnight render check.
  • Integration is the product. No single component is novel; the value is four systems joined exactly at the decision point.
  • Same method, different domain. Constraints first, real users (a household), and iteration in production. The approach that ships editorial tools ships this too.

Back to the studio problem

  • One screen over five silos. Licence portal, farm manager, kit spreadsheet, renewal calendar, reseller quotes: the dashboard's polling-and-normalising pattern joins exactly that shape of scattered state, so "what's low, where" stops being a five-tab question.
  • The overnight agent is a stock clerk. A nightly scan that matches prices and thresholds against a wants list is the same job as watching renewal dates and seat counts; surprises become flagged decisions waiting in a morning queue.
