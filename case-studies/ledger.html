Ledger

SC BUDGETTK 12FPS 24TC 01:23:41:12
ROLE Product, design, engineering direction STACK Next.js · Supabase · Vercel STATUS In production since May 2026

A full-stack, real-time budgeting app for two people and one household, in daily use within a week.

The studio problem

Every show runs on a cost report that's already out of date. Departments burn bid-days that only surface at month end; overages on awarded work get discovered after the burn, not during it; and when two entities front costs for each other (a facility and its outpost, a show and its reshoot unit), reconciling who owes whom is a spreadsheet ritual nobody trusts. Line producers don't need more reports — they need actuals that move in real time, visible to everyone who can spend.

Shrink that problem until it fits in a kitchen: it's still real money, real cost centres, and two people who both spend against the same budget. That's the version I could build for, live in production, with users who complain immediately.

The same problem at household scale

Two people, one shared household, two credit cards, and a month-end ritual of copy-pasting PocketSmith data into a spreadsheet. It was hard to see which card was used at entry time and harder to track who paid us back. The paid options (YNAB, PocketSmith premium) had too much overhead or missed the exact feature set: shared real-time entry, household-level budgets, per-category alerts.

Constraints set before building

Two users, one householdAll data belongs to the household; a partner's entries appear instantly.
Real-time syncBoth people see the same numbers without refreshing. That's the whole point.
No bank syncManual entry keeps spending intentional and removes the security overhead of connected accounts.
Mobile-first PWABoth users live on phones; the app installs to the home screen.
Free to runVercel and Supabase free tiers cover two users indefinitely.

How it was built: twelve sessions

I have scripting experience (Python, VFX pipeline tooling) but had never built a full-stack web app, written row-level security policies, or set up CI/CD. Every technical decision was made through conversation with Claude: I described what the household needed; the architecture, schema, components, and bug fixes came out of the dialogue. I reviewed, tested, and steered.

Session 0–1Design references and scoping. Data model, tech stack, and documentation decided before a single line of code.
Session 2Supabase schema deployed: 7 tables, RLS policies, realtime subscriptions. Login, dashboard, transaction entry, and live sync working the same day.
Session 3Six features in one sitting: PWA install, category drill-down, budget alerts, custom categories, CSV export, and a hand-rolled SVG donut chart. No chart library.
Session 4Dark mode: a full CSS custom-property token system, three-way theme preference, and a Python bulk migration that removed every hardcoded hex in one pass. My wife onboarded that night.
Sessions 5–6Per-card tracking (three cards, colour badges everywhere), recurring bills page with 27 bills seeded, keyword auto-matching, and smart suggestions that came from watching her actually use it.
Sessions 7–9People page with per-person net spend, payback tracking (who fronted what, what came back), and a two-page PDF report designed visually in Claude Design before implementation.
Session 10Nine features in one day, including a monthly close system with locked months and a jsPDF export that survives iOS Safari.
Sessions 11–12Mobile UI polish across every view, then a pure product-design session: a consent-based budget-adjustment system scoped, schema migration written, no code. A meeting, a design doc, and a review in one conversation.

Hours per session, not weeks — roughly 8,000 lines deployed. In production since 2026-05-13 with two daily users, and still growing. Sprint reviews and architecture calls ran through a multi-tier agent protocol: one agent strategises, another breaks work into surgical tasks, a third executes, and consultant agents advise into a shared think-tank log. The review that took Ledger from 8/10 to 9/10 also deleted 682 lines of dead code.

The dark mode story

Dark mode wasn't in the spec; it emerged because a budgeting PWA gets used in bed at night. The catch: the whole codebase used inline hex colours, so theme classes couldn't work without a rewrite. The fix was a design-token system plus a Python migration script that handled every quote variant in a single pass across eight files. Zero hardcoded palette values remain; the theme switches with one attribute. The pattern was worth keeping, so it's documented as a standalone concept in my knowledge base.

Design is part of the stack

Pages with real visual complexity (the history page's radial donut, the two-page PDF report) were designed in Claude Design first: it produces a self-contained HTML mockup with real colours and real data structures, iterated until right, then handed to the coding session as the spec. One person, one toolchain, no designer-to-engineer handoff loss. That two-step workflow is now how every visually heavy feature gets built, including parts of this site.

What it demonstrates

  • Agentic engineering scales past scripts. Managed database, auth, realtime subscriptions, and deployment pipelines are all reachable through disciplined conversation plus domain judgment.
  • Constraints first, code second. The five constraints above ruled out every off-the-shelf option and picked the stack before a line was written.
  • Real users are the bar. The app replaced a genuinely painful workflow for two people who use it every day.
  • The best features were discovered, not planned. Payback tracking, the smart bill suggestions, and the token-based dark mode all emerged from watching real use. The spreadsheet never could say who owed what; Ledger closes that loop with per-person net spend and a paybacks table in the PDF report.

Back to the studio problem

  • Live actuals are buildable in-house. Realtime entries against shared budgets, per-spender attribution, and locked months are exactly the cost-report mechanics a show needs; the household version proves the pattern end to end on free-tier infrastructure.
  • Reconciliation stops being a ritual. Ledger's payback tracking is the two-entities-fronting-costs problem in miniature: every fronted cost carries who paid, and the month-end report nets it out automatically instead of by spreadsheet archaeology.
