Dental Catalog

A patient photo manager for a clear-aligner dentist: browse by patient, compare sessions side by side, and sort new photos into clinical labels. Everything runs from one self-contained HTML file on her own drive, and it's in daily clinical use.

ROLE Product direction, domain translation STACK Single HTML file · File System Access API STATUS v2 in daily use

The problem

My wife photographs clear-aligner patients at every session: front, sides, occlusal, smile, shade guides, across a folder of 100+ patients going back years. There was no way to browse by patient or compare sessions without hunting through Windows Explorer. The paid apps are subscription-based and sync to a cloud backend. Sensitive patient data plus a tight budget ruled both out.

Constraints set before building

LightweightLives on a single drive, runs from any browser. Portable by construction.
SecureZero data leaves the device: it works entirely without a backend, an account, or a subscription.
ModularAs the patient folder grows, the tool grows with it, without ever requiring a migration.

Those three constraints eliminated every off-the-shelf option and pointed directly at a single self-contained HTML file using the File System Access API.

How it was built

I didn't write a line of code. I stated the constraints and dropped in five patient folders as a proof of concept. The model recognised the dental photo naming conventions unprompted, inferring that front / right / upper mapped to specific clinical views, and proposed a labelling scheme matching how dentists actually work. From there, iteration through prompting:

  • Iteration 1: core browser with patient grid, session timeline, gallery.
  • Iteration 2: clinical notes and status per session, persisted to both localStorage and a notes file in the patient folder.
  • Iteration 3: before/after comparison with a drag slider, the standard affordance in dental software.
  • v2 overhaul: full design rewrite plus a Sort Wizard with image fingerprinting. It samples pixel luminance via the Canvas API and suggests labels for new photos based on visual similarity to already-sorted ones.

2,313 lines shipped. Zero written by hand. Built in hours across two sessions. It replaced a painful manual workflow for a non-technical daily user, which is the only bar that matters.

What it demonstrates

  • Domain expertise drove the outcome, not coding expertise. I knew the problem and the constraints; the AI knew how to build. Neither alone ships this.
  • Iterative prompting compounds. The v2 image-fingerprinting Sort Wizard was unthinkable in week one; it became obvious only after watching real use of v1.
  • Zero ongoing cost. There is no infrastructure or subscription to maintain, so the tool costs nothing to keep running.
