Dental Catalog

SC SMILETK 22FPS 24IN 00:00:00:00 OUT 02:07:19:03
ROLE Product direction, domain translation STACK Single HTML file · File System Access API STATUS v2 in daily use

A patient photo manager in one self-contained HTML file: no cloud, no subscription, in daily clinical use.

SLATE PROBLEM BUILD PROOF NEXT

The studio problem

Every facility owns a growing library of versioned imagery: plates, WIP renders, comp versions, reference stills, stacked up per shot across years of shows. Supervisors review by wiping between versions; coordinators label incoming turnover (plate, ref, grade, QC frame) by hand; and finding "the approved version from March against today's WIP" means archaeology in a file browser. The review tools that do this well are seat-licensed and cloud-tethered, which client security (TPN-style) and budget both push back on.

The same shape at consumer scale: one practitioner, one growing image library, version-over-time comparison as the core workflow, and data that must never leave the building.

The same problem at clinical scale

My wife photographs clear-aligner patients at every session: front, sides, occlusal, smile, shade guides, across a folder of 100+ patients going back years. There was no way to browse by patient or compare sessions without hunting through Windows Explorer. The paid apps are subscription-based and sync to a cloud backend. Sensitive patient data plus a tight budget ruled both out.

Constraints set before building

LightweightLives on a single drive, runs from any browser. Portable by construction.
SecureZero data leaves the device: it works entirely without a backend, an account, or a subscription.
ModularAs the patient folder grows, the tool grows with it, without ever requiring a migration.

Those three constraints eliminated every off-the-shelf option and pointed directly at a single self-contained HTML file using the File System Access API.

How it was built

I didn't write a line of code. I stated the constraints and dropped in five patient folders as a proof of concept. The model recognised the dental photo naming conventions unprompted, inferring that front / right / upper mapped to specific clinical views, and proposed a labelling scheme matching how dentists actually work. From there, iteration through prompting:

  • Iteration 1: core browser with patient grid, session timeline, gallery.
  • Iteration 2: clinical notes and status per session, persisted to both localStorage and a notes file in the patient folder.
  • Iteration 3: before/after comparison with a drag slider, the standard affordance in dental software.
  • v2 overhaul: full design rewrite plus a Sort Wizard with image fingerprinting. It samples pixel luminance via the Canvas API and suggests labels for new photos based on visual similarity to already-sorted ones.

2,313 lines shipped. Zero written by hand. Built in hours across two sessions. It replaced a painful manual workflow for a non-technical daily user, which is the only bar that matters.

What it looks like, feature by feature

Recreations built from the tool's own design language. No real patients: representative data only.

The approach: quick tool first, foundation second

Before this project I had already proven the pattern on internal studio tooling: when a recurring "what's actually in this folder?" question was costing minutes every time, the answer was a two-minute tool — a folder walk plus a smart summary, about a hundred lines. Months later, a bigger visual inspection tool didn't start from scratch: it inherited the folder-walking, the naming-convention parser, and the caching from the quick tool, and added playback and a browsing UI on top.

Dental Catalog followed the same arc deliberately. v1 was the smallest thing that answered the daily question (browse by patient, compare sessions). Because v1 was built cleanly, the v2 overhaul — full redesign plus the fingerprinting Sort Wizard — reused its bones instead of replacing them. Build the quick tool right and it becomes the foundation; every expansion after it gets faster.

What it demonstrates

  • Domain expertise drove the outcome, not coding expertise. I knew the problem and the constraints; the AI knew how to build. Neither alone ships this.
  • Iterative prompting compounds. The v2 image-fingerprinting Sort Wizard was unthinkable in week one; it became obvious only after watching real use of v1.
  • Zero ongoing cost. There is no infrastructure or subscription to maintain, so the tool costs nothing to keep running.

Back to the studio problem

  • Version review without the seat licence. The session timeline and wipe compare are the supervisor's version-over-time workflow, running local-first — exactly the shape TPN-style client security asks for and cloud review tools can't offer.
  • Labelling on arrival, not by hand. The Sort Wizard's fingerprint-and-suggest pattern is the coordinator's turnover-labelling job: incoming media classified against what's already filed, confident matches batched, uncertain ones surfaced for a human call.
