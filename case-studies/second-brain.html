The studio problem

Onboarding is the same conversation on repeat. Every new artist asks where the LUTs live, what the naming convention is, why the pipeline does that one weird thing; every answer lives in a senior's head, a Slack scrollback, or a wiki nobody has updated since the last show wrapped. When that senior leaves, the knowledge wraps with them. And now there's a second audience with exactly the same problem: AI agents need context the same way new hires do — stated once, kept current, findable.

I built the household version: one vault that every agent session and every human question runs through.

How it's structured

The wiki is the truthPages for every entity, project, and concept, all catalogued in a master index. If it isn't in the index, it doesn't exist yet — a logged gap beats an invented answer.
Hard rulesRaw sources are immutable. Pages are never deleted, only superseded or archived. Every write updates the index and appends to an operations log. The audit trail can't lie.
Ingest, don't dumpNew material (a dev log, a URL, a day's notes) goes through a discuss-then-file workflow: takeaways agreed first, then filed, indexed, and logged.
Status vocabularyEvery page carries a controlled status (stub → developing → mature → superseded…), so staleness is visible instead of silent.
Verify, don't guessBefore an agent acts on an assumption, it runs a small read-only check and records the result as confirmed or inferred. A belief that was never tested doesn't get to drive a build.

Agents as the operating staff

The vault isn't just storage; it's the context layer a team of agents runs on. A tiered agent protocol defines who may do what: a strategy tier that never writes code, consultant tiers that advise into a shared think-tank log, an execution tier with surgical instructions and hard limits. Each project folder carries its own charter telling an incoming agent how to think there — the exact document you'd hand a new hire, except it's read on every session start. Onboarding cost: once.

The loop that makes it hold

Storage alone doesn't make a stateless agent reliable; the discipline around it does. Five layers turn a fresh session into something that behaves like a colleague who never forgets: a constitution every session reads first, the memory of the wiki and its logs, a navigation index that points to the right page fast, a verification habit that checks an assumption before acting on it, and a ratchet — every session ends with a handoff so the next one resumes instead of re-deriving.

The verification layer is the one that earns the most trust. A day's work usually opens by turning inferred beliefs into checked facts: a small read-only probe confirms a field name, a permission, an assumption baked into an old note — and only then does anything get built on top of it. Kept over time it becomes a track record: a running count of checks with no misses, because a guess never shipped as if it were a fact.

It compounds the more you probe

Most notes rot: written once, never revisited, quietly drifting out of date. This vault does the opposite, because using it is what maintains it. Every probe either gets answered from the wiki or exposes a gap, and a gap gets discussed and filed, so the next question on that thread lands warm. Ask about an area enough times and it stops being a lookup and becomes a cluster: a mature page cross-linked to everything around it, so an agent starts by reading the whole neighbourhood instead of one file.

That's the compounding. What started as a handful of stubs is now dozens of linked pages, and each answer arrives faster and better-connected than the last. The more it's probed, the more it's worth — the exact opposite of a wiki nobody has touched since the last show wrapped.

The discipline agents run on

The vault is the context; a set of working disciplines is what keeps agent sessions trustworthy once that context is loaded. The query mock above already shows two of them in action: an answer is never given without a "where this lives" citation, and a claim is labelled by how it was actually established, not stated as settled fact. The same disciplines apply beyond the vault: a live system's behaviour gets checked before it's acted on rather than assumed from memory; a suspected bug gets isolated with a small, deterministic repro before production code changes; a fix stays scoped to the exact case that's broken instead of applied everywhere "to be safe"; and every write-up follows the same shape, so a reader — human or agent — can resume cold without re-deriving anything.

What it demonstrates

  • Context is infrastructure. The reason my agent sessions ship working software quickly is that they start warm: constraints, history, and conventions are one read away, not re-explained per session.
  • Conventions beat willpower. Never-delete, always-log, discuss-before-filing: the vault stays trustworthy because the rules are structural, the same way a studio's publish pipeline keeps artists from overwriting each other.
  • Verification is a feature. Checking an assumption before building — and tagging what's confirmed versus inferred — is why the builds ship clean: nothing is constructed on a belief that was never tested, and an honest "inconclusive, deferred" beats an invented answer.
  • It compounds. HQ is its control panel, the case studies on this site were drafted from its pages, and every tool documented here made the next tool faster to build. A studio wiki run this way would do the same for every new hire and every agent the studio ever onboards.

Back to the studio problem

  • Onboarding becomes a read, not a conversation. A per-project charter that every new hire and every agent session reads on arrival is the structural fix for the same-questions-on-repeat problem; the senior's knowledge survives the senior leaving.
  • Trust is enforced, not hoped for. Never-delete, always-log, cite-your-source: the conventions that keep this vault honest are the same ones that would keep a facility wiki current past the end of the show that wrote it.
