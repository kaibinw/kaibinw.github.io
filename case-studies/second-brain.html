Second Brain

A markdown knowledge vault with hard conventions, operated by agents. Every project, decision, and dev session gets ingested into a wiki that both humans and AI can read — so no context is ever explained twice, to a person or to a machine.

The studio problem

Onboarding is the same conversation on repeat. Every new artist asks where the LUTs live, what the naming convention is, why the pipeline does that one weird thing; every answer lives in a senior's head, a Slack scrollback, or a wiki nobody has updated since the last show wrapped. When that senior leaves, the knowledge wraps with them. And now there's a second audience with exactly the same problem: AI agents need context the same way new hires do — stated once, kept current, findable.

I built the household version: one vault that every agent session and every human question runs through.

How it's structured

The wiki is the truthPages for every entity, project, and concept, all catalogued in a master index. If it isn't in the index, it doesn't exist yet — a logged gap beats an invented answer.
Hard rulesRaw sources are immutable. Pages are never deleted, only superseded or archived. Every write updates the index and appends to an operations log. The audit trail can't lie.
Ingest, don't dumpNew material (a dev log, a URL, a day's notes) goes through a discuss-then-file workflow: takeaways agreed first, then filed, indexed, and logged.
Status vocabularyEvery page carries a controlled status (stub → developing → mature → superseded…), so staleness is visible instead of silent.

Agents as the operating staff

The vault isn't just storage; it's the context layer a team of agents runs on. A tiered agent protocol defines who may do what: a strategy tier that never writes code, consultant tiers that advise into a shared think-tank log, an execution tier with surgical instructions and hard limits. Each project folder carries its own charter telling an incoming agent how to think there — the exact document you'd hand a new hire, except it's read on every session start. Onboarding cost: once.

What it demonstrates

  • Context is infrastructure. The reason my agent sessions ship working software quickly is that they start warm: constraints, history, and conventions are one read away, not re-explained per session.
  • Conventions beat willpower. Never-delete, always-log, discuss-before-filing: the vault stays trustworthy because the rules are structural, the same way a studio's publish pipeline keeps artists from overwriting each other.
  • It compounds. HQ is its control panel, the case studies on this site were drafted from its pages, and every tool documented here made the next tool faster to build. A studio wiki run this way would do the same for every new hire and every agent the studio ever onboards.
